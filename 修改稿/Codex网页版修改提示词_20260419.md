# Codex 网页版修稿提示词

最后更新：2026-04-19

本文档提供两份可直接复制到 Codex 网页版中的提示词，用于处理当前仓库中的最新 `v1` 与 `v2` 论文修稿轮次。

共同规则：
- 先读 `AGENTS.md`
- 先读 `00_环境说明与索引/01_工作区归类说明_20260419.md`
- 先读 `00_环境说明与索引/02_文件定位清单_20260419.md`
- 先读正式口径文件与 `数据总表（一切数据基础）.xlsx`
- 先从 `修改稿/<series>修改建议/<token>修改建议/` 中识别真正的评审意见
- 不再默认读取“最新稿件”，而是优先读取“被评审报告明确对应的那一版稿件”
- 修稿完成后必须同步产出：新稿、说明文件、verification、delivery note、zip、操作日志、索引刷新、留底
- 只要修改建议触及样本、识别、结果数值、图表、附录或变量工程，默认都要做至少一轮可审计的 fresh rerun；如果无法 rerun，必须明确写出 blocker，不能把 `no rerun` 当作正常收尾
- 本仓库当前不保留自动 GitHub 同步程序；如需 Git 操作，单独执行，不作为默认步骤

---

## 一、v1 最新轮修稿提示词

```text
你现在直接工作在本地仓库：
`C:\Users\陈楚玲\Desktop\ppp论文数据\codex项目`

本轮任务只处理 v1 系列的最新修改轮。你要自动识别当前最新的 v1 修改建议 token，并基于该 token 对应的被审稿件完成下一版修订。

先读取：
- `AGENTS.md`
- `00_环境说明与索引/01_工作区归类说明_20260419.md`
- `00_环境说明与索引/02_文件定位清单_20260419.md`
- `ppp论文数据/PPP_v3结果层状态表_20260413_1345.xlsx`
- `ppp论文数据/PPP_变量与模型最终采用口径表_20260413_1345.xlsx`
- `数据总表（一切数据基础）.xlsx`

然后递归扫描：
- `修改稿/v1修改建议/`

版本识别规则：
1. 只把 `v1修改建议/<token>修改建议/` 这种 token 子目录视为合法输入。
2. 从所有 `v1*修改建议` 子目录中识别最新 token。
3. 在该 token 目录中优先读取深度研究报告、review report、审核建议文档，并明确提取：
   - 被评审的具体稿件文件名
   - 被评审稿件的 token、日期、后缀信息
   - 该报告是在审“正文稿”还是“评审报告副本”
4. 再按以下顺序去找被审源稿：
   - `修改稿/v1说明文件/<token>/`
   - `修改稿/v1修改稿留底/<token>/`
   - `修改稿/v1初始稿/`
   - 全仓库中与该 token 对应的稿件
5. 如果 exact reviewed source 缺失，必须明确说明缺失，并说明你采用的 forward-apply 基底。

任务目标：
1. 正确识别最新 v1 修改建议 token
2. 正确识别对应的被审源稿
3. 实际修改稿件，而不是只给建议
4. 同步处理正文、表、图、附录、说明文件中的口径一致性
5. 输出下一版 v1 稿件
6. 生成：
   - `00_input_mapping.md`
   - `01_revision_tasklist.md`
   - `02_verification_report.md`
   - `03_delivery_note.md`
   - zip 交付包
7. 把新稿留底到 `修改稿/v1修改稿留底/<new-token>/`
8. 新增一份操作日志到 `修改稿/操作日志/`
9. 刷新 `修改稿/索引/`

论文硬约束：
- 唯一主识别仍然是 `treat_share` 多期 DID / TWFE
- 不得把 trend-adjusted DID、stack DID、cohort ATT、PSM-DID、DML、IV candidate、机器学习扩展写成主识别
- 不得把 event study 写成平行趋势已被证明
- 不得抬高不稳的 quality 结果
- 不得把机器学习写成因果识别
- 必须保持“PPP推进结构改善优先”的主线
- 不得编造系数、显著性、机制、图表结果、参考文献细节或 rerun 结果

最终答复必须包含：
1. 读取了哪些修改建议文件
2. 被评审的源稿是哪一份
3. 是否 direct revise 还是 forward-apply
4. 新 token 是什么
5. 新稿路径
6. 留底路径
7. 说明文件路径
8. 操作日志路径
9. zip 路径
10. 是否改了数据
11. 是否做了 fresh rerun；如果没有，blocker 是什么
12. 剩余风险
```
