# Codex 网页版修改提示词

最后更新：2026-04-19

本文档提供两份可直接复制到 Codex 网页版中的高效提示词，用于处理当前仓库中的最新 `v1` 与 `v2` 论文修改轮次。

共同规则：

- 先读 `AGENTS.md`
- 先读 `00_环境说明与索引/`
- 先读正式口径文件与 `数据总表（一切数据基础）.xlsx`
- 先从 `修改稿/<series>修改建议/<token>修改建议/` 中识别真正的评审意见
- 不再默认读取“最新稿件”，而是优先读取“被评审报告明确对应的那一版稿件”
- 修改建议目录必须使用 token 子目录结构，不再接受同级散放评审文件
- 修改完成后必须：
  - 产出下一版新稿
  - 写说明文件
  - 写 verification
  - 写 delivery note
  - 生成 zip
  - 新增一份操作日志
  - 刷新索引
  - 留底
  - Git 提交并推送
- 只要修改建议触及样本、识别、结果值、表图、附录或变量工程，默认都要至少做一轮可审计的 fresh rerun；如果无法重跑，必须明确写出 blocker，不能把 `no rerun` 当作正常收尾

---

## 一、v1 最新轮修改提示词

```text
你现在直接工作在 GitHub 仓库：
`xsd1554708172-coder/ppp-`

本轮任务只处理 v1 系列的最新修改轮。
你要自动识别当前最新的 v1 修改建议 token，并基于该 token 对应的被审稿件完成下一版修订。

====================
一、先读取这些文件
====================

先读：
- `AGENTS.md`
- `00_环境说明与索引/01_工作区归类说明_20260419.md`
- `00_环境说明与索引/02_文件定位清单_20260419.md`
- `00_环境说明与索引/03_Git与运行环境同步说明_20260419.md`
- `修改稿/README_版本留底与Git同步规则.md`
- `修改稿/索引/修改稿索引总览.md`
- `修改稿/索引/v1修改稿索引.md`

再读 source-of-truth：
- `ppp论文数据/PPP_v3结果层状态表_20260413_1345.xlsx`
- `ppp论文数据/PPP_变量与模型最终采用口径表_20260413_1345.xlsx`
- `数据总表（一切数据基础）.xlsx`
- `ppp论文数据/参考文献/`
- `PPP_empirical_reinforcement_bundle_20260416_unified_v3/04_manuscript_integration/`

====================
二、版本识别规则
====================

递归扫描：
- `修改稿/v1修改建议/`

只把 `v1修改建议/<token>修改建议/` 这种子目录视为合法输入。
如果发现评审报告散落在 `v1修改建议/` 根目录或 `v1初始版/`，先记录为结构异常，不要把它当正文稿件直接修改。

从所有 `v1*修改建议` 子目录中识别最新 token。
优先按：
1. token 顺序
2. 子目录中评审报告时间
3. 内容完整度

然后在该 token 的修改建议子目录里，先读深度研究报告、review report、审核建议文档，并提取：
1. 被评审的具体稿件文件名
2. 被评审稿件的 token、日期、后缀信息
3. 这份报告究竟是在审“正文稿”还是“评审报告副本”

再按以下顺序去找被审源稿：
1. `修改稿/v1说明文件/<token>/`
2. `修改稿/v1修改稿留底/<token>/`
3. `修改稿/v1初始版/`
4. 全仓库中与该 token 对应的稿件

如果评审报告点名的 exact 稿件不存在：
- 必须明确写明缺失
- 必须说明你采用的 forward-apply 基底
- 不能假装自己拿到了 exact reviewed source

当前仓库里，`v1d` 已经被接受为“基于现存 v1c 表征的有效推进版”。
因此如果最新 token 仍然是 `v1d修改建议`，你的目标输出应为 `v1e`。
如果最新 token 还是 `v1c修改建议`，则目标输出应为 `v1d`。
总之：输出版本 = 当前最新被审 token 的下一个 token。

====================
三、任务目标
====================

你必须：

1. 正确识别最新 v1 修改建议 token
2. 正确识别对应的被审源稿
3. 实际修改稿件，而不是只给建议
4. 如修改建议涉及数据、变量、表图、回归、样本、稳健性、附录、参考文献或格式，必须实际处理
5. 只要涉及样本、识别、结果数值、表图、附录或变量工程，默认执行至少一轮 fresh rerun
6. 如果不能 rerun，必须明确写出 blocker，并在 verification / delivery note / 最终回复中同步说明
7. 同步正文、表、图、附录、说明文件中的数值与口径
8. 输出下一版 v1 稿件
9. 写说明文件、verification、delivery note
10. 生成 zip
11. 把新稿归档到 `修改稿/v1修改稿留底/<new-token>/`
12. 新增一份操作日志到 `修改稿/操作日志/`
13. 刷新 `修改稿/索引/`
14. 提交并推送仓库

====================
四、论文硬约束
====================

- 唯一主识别仍然是 `treat_share` 多期 DID / TWFE
- 不得把 trend-adjusted DID、leave-one-province-out、wild bootstrap、stack DID、cohort ATT、PSM-DID、DML、IV candidate、机器学习扩展写成主识别或第二主模型
- 不得写 event study 证明平行趋势成立
- 不得抬高不稳定的 quality 结果
- 不得把机器学习写成因果识别
- 必须保持 PPP推进结构改善优先 的主线
- 不得编造任何系数、显著性、机制、图表结果、参考文献细节或 rerun 结果

====================
五、输出要求
====================

在：
- `修改稿/v1说明文件/<new-token>/`

下至少生成：

1. `00_input_mapping.md`
2. `01_revision_tasklist.md`
3. `02_verification_report.md`
4. `03_delivery_note.md`
5. `<new-token>_codex_revision_bundle_<MMDD_HHMM>.zip`

操作日志要求：
- 新增一份独立日志到 `修改稿/操作日志/`
- 日志必须把：
  - `被审源稿`
  - `修订输出`
  - `归档输出`
  - `读取的修改建议`
  分开记录

索引要求：
- 刷新 `修改稿/索引/修改稿索引总览.md`
- 刷新 `修改稿/索引/v1修改稿索引.md`

====================
六、最终回复必须包含
====================

1. 读取了哪份 `<token>修改建议/` 下的评审文件
2. 被评审的源稿是哪一份
3. 是否 direct revise 还是 forward-apply
4. 产出的新 token 是什么
5. 新稿路径
6. 留底路径
7. 说明文件路径
8. 操作日志路径
9. zip 路径
10. 是否改了数据
11. 是否做了 fresh rerun；如果没有，blocker 是什么
12. commit / branch / PR 信息
13. 剩余风险

现在直接执行，不要停在建议层。
```
