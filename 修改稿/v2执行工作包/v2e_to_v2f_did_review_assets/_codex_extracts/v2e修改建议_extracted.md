v2e修改建议

基于当前稿件、仓库数据资产、压缩包 DID 参考文献与官方政策文件的审稿式修订方案

用途说明：本报告针对 v2e_0420_1944 当前版本，重点回答“是否适合继续用 DID 补强、如何写得更稳、下一步最值得补哪一块证据”。

一、总判断

结论先行：这篇稿子当前最需要的不是“再加一个 DID”，而是把已经存在的 treat_share 多期 DID/TWFE 识别链条做硬、做实、做边界化。就你当前 v2e 的材料看，论文已经有 DID 框架，但仍处于“可发表的谨慎版关联识别”而不是“高说服力准自然实验版 DID”的状态。

如果沿当前仓库和稿件口径直接投稿，我的建议不是新增普通 DID、PSM-DID 或合成控制，而是：保留 treat_share 为唯一主识别；统一改写为连续处理强度下的平均条件关联；把事件研究降格为诊断；把 A/B/C/D 改写为政策文本证据线索；把真正的机制补强转到非文本源变量。

如果你的目标是明显提升识别强度，则最值得投入的不是继续堆方法名，而是恢复真实 city-year treatment registry，把当前省级暴露强度回溯到可核的城市名单与首次处理年，再升级为 city-year stacked DID / Sun-Abraham 事件研究。

当前最佳策略：不新增普通 DID，不把 PSM-DID/SCM 拉到主文；把现有 treat_share 模型写实、写稳、写边界。

当前可接受定位：连续处理强度下的多期 DID/TWFE 面板识别，结论解释为平均条件关联。

当前最值得投入的升级：恢复真实 city-year treatment registry，然后升级为 city-year stacked DID / modern event study。

四、最核心的修改建议：不是再加 DID，而是把当前 DID 做硬

第一，把 treat_share 从“结果附近构造的暴露强度”尽可能恢复为“真实城市名单 + 首次处理年 + 聚合权重”的上卷结果。当前 implied city-slot registry 只能支持继续分析，不能支持高强度因果表述。

第二，把事件研究彻底降格为诊断工具。你现在最稳妥的写法不是‘平行趋势成立’，而是‘事件研究揭示了动态路径与识别边界，因此本文进一步采用趋势调整、小样本推断与逐省剔除进行防守性检验’。

第三，把主结果聚焦到 exec_share / proc_share / log-ratio。不要试图把 ppp_quality_zindex 抬成中心结论。

第四，把 A/B/C/D 从‘机制检验’改成‘政策文本证据线索’，并尽快补独立来源机制变量。

第五，把 share 类因变量的统计处理做得更规范：保留 baseline share 回归，但至少增加 log-ratio 与 fractional response 作为稳健性。

第六，把论文贡献重新收束为：数字化改革首先与 PPP 推进结构重组相关，而不是立即全面提升治理质量。这个收束反而会让文章更可信。

六、如果你愿意做一轮真正意义上的升级：建议的理想设计

理想升级版不是在省级 31 个单元上继续堆更多 DID 名称，而是把处理回到城市层。具体做法是：基于真实的 city-year 改革名单，识别各城市首次进入数字政务改革的年份；把城市作为处理单位，未处理或尚未处理城市作为对照；将 PPP 项目推进结果下沉到 city-year 或 project-year 层级；使用 stacked DID / Sun-Abraham 事件研究估计动态效应。

若你能拿到 PPP 项目级的采购启动时间、签约时间、执行启动时间、退库时间，那么比 share 回归更强的方法是 duration / hazard model。因为你的研究问题本质上是‘数字化改革是否改变了项目从采购走向执行的转换速度’，这比省级份额更直接，也更能回应公共管理审稿人的问题意识。

若未来能同时获得非 PPP 的政府投资项目或其他公共采购项目，还可以构造 DDD：PPP 项目 × 数字改革暴露 × 政策后。这样能更有力地证明你识别到的是 PPP 场景特有的治理重组，而不是一般性的行政效率改善。

七、可直接写入论文的方法设计文本（按当前可行版本）

制度背景

本文研究的制度背景应分成两条线并明确区分。第一条线是数字政务改革：2016 年国务院印发《关于加快推进“互联网+政务服务”工作的指导意见》，明确提出推进数据共享、优化服务流程、建设一体化网上政务服务平台；2018 年国务院印发《关于加快推进全国一体化在线政务服务平台建设的指导意见》，强调加快建设全国一体化在线政务服务平台、全面推进“一网通办”；2022 年国务院又就政务服务标准化、规范化、便利化作出系统部署。第二条线是 PPP 制度环境：2014 年财政部发文推广 PPP 模式，2017 年财政部规范 PPP 项目库管理并强调防止 PPP 异化为融资平台，2023 年又转入 PPP 新机制。数字政务改革是本文要识别的主冲击，PPP 政策是结果发生的制度环境与潜在干扰源，而不是本文的主处理。

识别策略

在现有数据条件下，本文不宜将 treat_share 回归解释为标准二元 DID 的处理平均效应。更准确的做法是：利用城市层数字化改革状态在省级层面的聚合份额，构造省—年层面的改革暴露强度 treat_share，并在地区和年份固定效应框架下估计数字化改革暴露强度与 PPP 推进结构之间的平均条件关联。由于多数省份首次处理年份集中在 2016 年、never-treated 地区较少，且事件研究中处理前系数并非完全平稳，本文将主结果界定为方向性、过程性的准因果证据，而不将其表述为无条件强因果效应。

模型设定

基准模型建议写为：Y_pt = β·treat_share_pt + γX_pt + μ_p + λ_t + ε_pt。其中，Y_pt 为省 p 在年份 t 的 PPP 推进结构或辅助治理结果；treat_share_pt 为由城市改革状态聚合形成的省级数字化改革暴露强度；X_pt 为控制变量；μ_p 和 λ_t 分别为地区固定效应与年份固定效应。为回应省际异质趋势问题，稳健性检验中进一步加入省份线性趋势。考虑到聚类数有限，标准误按省聚类，并同步报告 wild cluster bootstrap 的 p 值。

变量定义

被解释变量应明确区分主结果与辅助结果。主结果为 exec_share 与 proc_share，分别表示执行阶段占比和采购阶段占比；为回应份额型因变量的加总约束，建议新增 log((exec_share+c)/(proc_share+c)) 作为构成性补强，其中 c 取最小正份额的一半。ppp_quality_zindex 仅作为辅助质量结果，不承担主结论。核心解释变量为 treat_share；did_any 与 did_intensity 仅作为替代口径。控制变量优先使用当前仓库中已经存在的数字经济、数字基础与创新能力变量，包括 dfi、digital_econ、base_station_density、software_gdp_share、it_service_gdp_share、ln_rd_expenditure、ln_tech_contract_value、ln_patent_grants，以及文本覆盖或缺失控制量。

数据处理

数据处理部分要把‘文档级文本—省年治理测度—省级主面板—项目级扩展底表’四层链条写清楚，并把 1472→1307→288→266→262 的筛选流程独立成图或附录表。最关键的是处理变量定义：必须说明 treat_share 由哪些城市在何年进入数字化改革后汇总而成；如果当前只能使用 implied city-slot registry，则正文必须明确其 proxy 身份，并把省级重构材料定义为复核资产而非完整复现包。

平行趋势检验

基于当前结果，事件研究不能被写成‘平行趋势已经成立’。建议写法是：事件研究用于考察动态路径并诊断识别边界；由于部分领先项存在显著性或边际显著性，本文不将其视为对平行趋势假设的强证明，而是进一步采用省份线性趋势、伪政策时间、随机置换和逐省剔除等方式评估主结果的稳健性。若未来恢复真实 city-year cohort，则可进一步采用 stacked DID 或 Sun-Abraham 事件研究进行更规范的动态识别。

稳健性检验

稳健性检验应服务于三类质疑：趋势偏误、构成性约束与小样本推断。第一，加入省份线性趋势，检验主结果是否仍保留方向和基本显著性；第二，新增 log-ratio、fractional response 或 compositional robustness，以回应 share 结果的机械约束；第三，采用 wild cluster bootstrap、leave-one-out、placebo year、随机置换处理时点和剔除特殊地区/年份，检验结论是否由有限聚类或单一区域驱动。PSM-DID、DML、候选 IV 等只作为附录诊断，不进入正文主防守层。

机制检验

机制部分不宜再写成独立的强中介识别。当前 A/B/C/D 与 treat_share 同样依托政策文本工程，因此它们更适合被解释为制度表达和治理重心变化的文本证据线索。真正的机制补强应来自非文本源变量，如审批时长、采购到执行间隔、公告披露完整度、项目退库率、在线服务指数、项目执行周期等。只有这些变量完成独立构造并与主面板可靠对接后，才能在正文中新增‘独立机制补强’段落。

结果解释

结果解释必须坚持边界控制。若 treat_share 对 exec_share 为正、对 proc_share 为负，应解释为：数字化改革暴露强度较高的地区更可能出现 PPP 项目由采购阶段向执行阶段的相对份额重组，说明改革首先与治理过程重组相关。不得进一步外推为‘数字化改革已经全面提升 PPP 治理质量’。对 ppp_quality_zindex，只能写作方向性信号而非中心结论。

局限性说明

局限性至少要写四点。第一，处理变量目前仍未完全回到真实城市名单与原始阈值底表，treat_share 更适合被理解为省级暴露强度而非标准处理指示。第二，多数地区处理时点集中在 2016 年，not-yet-treated 信息有限。第三，事件研究领先项并非完全平稳，因而主结果仍需理解为有限条件下的准因果证据。第四，主结果变量为结构性份额指标，不能完全替代项目阶段转换速度和持续时间本身。未来研究应进一步恢复城市层处理名单、引入项目级时间戳，并用 duration / hazard model 直接识别采购到执行的转换过程。

九、最终建议：你的文章应当怎样改

如果目标是尽快形成一版更稳的投稿稿：保持当前 treat_share TWFE 为唯一主识别，明确写成平均条件关联；把主发现压缩到 PPP 推进结构重组；加强趋势调整、wild bootstrap、log-ratio 和 leave-one-out；机制部分降格为文本证据线索。

如果目标是把识别说服力明显再上一个台阶：优先重建 city-year treatment registry，而不是继续做更多省级回归；这一步的收益远高于再加一项 PSM-DID 或再画一张事件研究图。

如果目标是把论文做成更强的公共管理研究：尽快转向项目级阶段转换的 duration / hazard design。这将比当前 share 结果更直接地回答‘数字化改革如何改变 PPP 推进过程’。
