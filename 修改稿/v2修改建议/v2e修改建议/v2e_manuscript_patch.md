# v2e Manuscript Patch

本文件只提供面向正文的 patch-style 修改建议，不直接覆盖任何 `.docx` 主文稿。除已由 v2d 或正式文件核实的数值外，所有待 rerun 内容均标注为“未重跑，不得当作结果”。

## Patch 1：第4.3节末尾新增 treatment registry 说明

### 替换/插入位置

放在第4.3节 `treat_share` 公式与样本流转说明之后。

### 建议插入段落

为增强处理变量的可审计性，v2e 版本将新增 city-year treatment registry 作为附录 A0。该 registry 的目标不是改变本文主识别变量，而是把 `treat_share` 的底层构造链条从“省年重构结果”进一步追溯到“城市名单—首次处理年份—年度处理状态—省年聚合权重”。具体字段至少包括 `province`、`city`、`city_id`、`year`、`first_treat_year`、`treated`、`treatment_source`、`source_note` 与 `aggregation_weight`。在该 registry 完成之前，本文不将当前附录A表述为完整城市级复现包；它只构成 province-year 层面的复核资产。

### 写作边界

- 不得说“城市级处理名单已经完整复现”，除非 registry 真实填充并审计通过。
- 不改变 `treat_share` 主识别地位。

## Patch 2：第4.4节主识别边界补强

### 替换/插入位置

放在主模型公式解释后、event study 边界说明前。

### 建议替换段落

与若干以外部政策批次或基础设施落地为处理来源的 DID 研究相比，本文的 `treat_share` 更接近连续暴露强度变量，而不是单一城市或企业层面的二元处理指示。因此，本文主识别的合理解释不是“某一城市是否受冲击”的标准 ATT，而是在地区固定效应、年份固定效应和统一控制变量条件下，省级数字化改革暴露强度与 PPP 推进结构之间的平均条件关联。v2e 的补强方向，是在不改变主识别框架的前提下，进一步公开 city-year registry 与省年聚合审计，使 `treat_share` 从 manuscript-facing 构造变量转化为可逐层追查的处理强度变量。

## Patch 3：第5.2节份额型结果段落补强

### 替换/插入位置

放在 log-ratio 补充估计段落之后。

### 建议插入段落

需要强调的是，执行阶段占比上升与采购阶段占比下降具有构成性联系，因此本文不把两者简单写作两个相互独立的结果。v2d 已经通过执行/采购对数比率提供一层构成性补强；v2e 进一步建议在附录中补充执行—采购差值、转换率或 fractional response / compositional robustness 的可行性检验。上述检验的作用是回答“推进结构重组是否只是份额约束下的机械变化”，而不是替代第5.2节的 `treat_share` TWFE 主结果。相关结果必须以 fresh rerun 为准；未重跑，不得当作结果。

## Patch 4：第5.4节机制段落升级为“文本线索 + 独立机制补强”

### 替换/插入位置

替换第5.4节开头对机制定位的说明。

### 建议替换段落

v2d 已将 A/B/C/D 维度降格为“政策文本证据线索”，这一处理应在 v2e 中继续保留。原因在于，A/B/C/D 与 `treat_share` 均依托政策文本工程，二者之间的关系不能直接解释为来源独立的机制识别。v2e 的机制升级方向，是在保留文本线索的同时，另行引入非文本源机制变量，例如审批时长、采购到执行间隔、公告披露完整度、项目退库率、在线服务指数或项目执行周期。只有当这些变量完成数据构造并与正式主面板或项目底表可靠对接后，才允许新增“独立机制补强”段落；否则第5.4节仍应维持为政策文本证据线索。

## Patch 5：第5.6节防御性稳健性总述

### 替换/插入位置

替换第5.6节开头总述段。

### 建议替换段落

本文的稳健性安排遵循“主识别—防守检验—边界诊断”三层结构。第一层仍是表4中的 `treat_share` 多期 DID / TWFE；第二层包括 trend-adjusted DID、leave-one-province-out 与 wild cluster bootstrap，分别回应省际异质趋势、单一地区驱动和有限聚类推断风险；第三层包括 cohort / stack / placebo / pseudo-treatment / randomization diagnostics 等边界诊断。后一组检验不构成新的主识别，也不用于抬升结论强度，而是帮助说明主结果在哪些设定下方向保持、在哪些设定下统计强度变弱。所有 v2e 数值必须以 fresh rerun 后的结果为准；未重跑，不得当作结果。

## Patch 6：第2章 DID 文献嵌入点

### 插入位置

第2章数字政府/政策冲击/PPP治理文献综述之后，方法与识别缺口之前。

### 建议插入段落

近期中文顶刊和准顶刊中的 DID 研究显示，可信的制度冲击研究通常并不依赖估计量堆叠，而依赖一条可审计的数据链：处理来自外部政策批次、行政批复、司法试点或基础设施落地，处理组与对照组划分可以追溯，处理时点可以核对，结果变量与机制变量尽量来自处理变量之外的数据源。国家级新区、新城新区、国家级大数据综合试验区、行政诉讼跨区划管辖、社保征管体制改革、三峡送电、破产审理方式改革试点等研究的共同经验是，主识别首先建立在原生处理名单和清晰时点之上，随后再通过动态检验、安慰剂、删样诊断、异质性与机制分析提供防守或解释。本文借鉴这一写法，但不改变自身以 `treat_share` 多期 DID / TWFE 为唯一主识别的结构。

## Patch 7：结论边界段落

### 替换/插入位置

放在结论中“推进结构改善优先”之后。

### 建议插入段落

因此，本文对数字化改革作用的解释应保持在“结构性改善”和“条件关联证据”的范围内。当前证据较稳定地支持 PPP 项目推进结构由采购阶段向执行阶段的相对重组，但尚不足以证明综合治理质量全面提升；事件研究提供的是动态路径与识别边界提示，而不是平行趋势已经完全成立的证明；A/B/C/D 文本维度提供的是制度表达和治理重心变化的线索，而不是来源独立的强机制识别。v2e 后续若完成 treatment registry、独立机制变量和文本效度审计，可以增强证据链的可审计性，但仍不应把防守检验写成新的主识别。

## 禁止写法复核

- 禁止：“证明 PPP 显著提升治理质量”
- 禁止：“完全验证平行趋势”
- 禁止：“机制已被强证实的表述”
- 禁止：“稳健性检验构成新的主识别”
- 禁止：“治理绩效全面改善的强表述”

## 2026-04-20 full rerun 后可用补丁（谨慎版）

> 本节基于 `v2e_rerun_result_snapshot.csv` 与 `rerun_logs/rerun_status_summary.md`。虽然 baseline、trend-adjusted DID、leave-one-province-out、wild cluster bootstrap 均已在当前环境退出码 0，但正文仍应以正式口径表为锚；audit rerun 仅作复核，不自动替换正式正文数值。

### Patch 5A：第5.6节 trend-adjusted DID 结果段落

在加入省份线性时间趋势后，`proc_share` 仍保持负向且在常规水平边际显著（coef = -0.352，p = 0.048），说明采购阶段占比下降并不完全依赖未调整的共同趋势设定。`exec_share` 的估计方向仍为正（coef = 0.226），但统计强度降至不显著（p = 0.194），因此只能写作“方向与主结果一致但对趋势设定敏感”。`ppp_quality_zindex` 在趋势调整后方向转为负且不显著（coef = -0.170，p = 0.678），应继续作为边界性结果处理，不得写成治理质量稳健改善。

### Patch 5B：leave-one-province-out 诊断段落

删一省诊断显示，三个结果变量均未出现相对正式基线方向翻转；但 `exec_share` 与 `proc_share` 各出现 1 次 5% 显著性状态变化，说明核心结构性结果的方向稳定性强于统计强度稳定性。该结果适合用于回应“单一省份驱动”的质疑，但不能写作新的识别策略。

### Patch 5C：wild cluster bootstrap 边界段落

在更保守的小样本聚类推断下，`exec_share` 与 `proc_share` 的方向仍与正式基线一致，但 wild cluster bootstrap p 值分别为 0.076 与 0.122，显示统计强度对有限聚类推断较敏感。`ppp_quality_zindex` 的 wild p 值为 0.237，继续不支持将治理质量口径抬升为主结论。正文可据此写作“防御性稳健性支持结构性改善的方向，但提醒统计强度边界”，不得写作“所有补充检验均强化主结果”。

### Patch 5D：baseline audit 与正式口径关系

本轮 audit rerun 与正式 5.3 anchor 的系数和 p 值非常接近但未达到 `1e-6` 完全一致容差：`exec_share` 正式 coef = 0.356、p < 0.001，audit coef = 0.360、p = 0.001；`proc_share` 正式 coef = -0.402、p < 0.001，audit coef = -0.407、p < 0.001；`ppp_quality_zindex` 正式 coef = 0.525、p = 0.213，audit coef = 0.523、p = 0.256。因此正文仍以正式口径表为准，audit rerun 只作为复核记录。
